# 开发规范 · 模拟民航客舱体感安全系统

> 原则:只写这个项目特有的、不约定就会出乱子的规范。
> 通用代码风格(命名、注释)一句"遵循阿里巴巴 Java 开发手册"带过即可,别展开注水。
> 真正有约束力的是下面这几条——它们是多人协作 / 设备后端对接不出错的前提。

---

## 1. MQTT Topic 命名规范 ★
【这是设备与后端的契约,必须全队统一,否则各发各的对不上。】
```
上行  cabin/{deviceId}/{type}      type ∈ {joystick, key, sensor}
下行  cabin/{deviceId}/cmd
回执  cabin/{deviceId}/ack
```
- deviceId 命名规则:【如 esp32-01 / w601-01】
- 新增 topic 必须先在接口文档登记,再写代码。

## 2. 消息 JSON 字段规范 ★
- 每条消息必带公共字段:`ts`(设备毫秒时间戳)、`seq`(单调递增)、`deviceId`。
- 字段命名统一小驼峰 / 下划线【二选一,定死】。
- 时间统一用毫秒时间戳 long,不传字符串日期。
- 枚举值(action / event_type / result)统一小写下划线,集中维护一份枚举清单,禁止散落魔法字符串。

## 3. 指令 / 回执的 ID 关联规范 ★★(本项目最关键的一条)
- 每条下行指令必须带全局唯一 `commandId`(UUID)。
- 设备回执必须**原样带回** `commandId`,后端据此对账。
- 没有 commandId 的指令一律视为非法,不下发。
- 这条是"指令回执对账"能成立的根基,直接支撑系统核心亮点,务必全员遵守。

## 4. 会话状态流转规范
- 状态迁移只能由后端的状态机统一驱动,**禁止前端/设备直接改会话状态**。
- 所有"失败迁移"必须记录原因(写入事件表 fail_reason)。
- 设备上报只代表"客观事实",对错判定一律由后端结合状态机做出,设备端不写任何业务判断逻辑。

## 5. 数据落库规范
- 实时流数据必须归属到具体的 `record_id`(某用户某次会话),禁止裸落库。
- 结算类写操作(积分/里程)必须幂等,禁止"结束模拟"重复执行导致重复结算。

## 6. 分支 / 协作规范(团队)
【你们几个人,简单定一下:主分支保护?功能分支命名?谁合谁的?——小团队几句话够了,别抄大厂流程。】

## 7. 环境 / 配置规范
- 敏感配置(MQTT 密码、数据库密码、AI 接口 key)不进 git,用配置文件 + .gitignore / 环境变量。
- 【EMQX 地址、端口等环境相关配置集中在 application.yml,不硬编码散落各处。】

## 8. 通用代码风格
遵循《阿里巴巴 Java 开发手册》。【一句话带过,不展开。】
