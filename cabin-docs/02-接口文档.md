# 接口文档 · 模拟民航客舱体感安全系统

> 本系统接口分三类:MQTT 上行(设备→后端)、WebSocket 下行(后端→前端)、HTTP(用户/会话事务)。
> 接口少是这个系统的真实情况,薄即完整,不要注水。
> 唯一要抠准的是「下行指令 + ACK 回执」这一对消息——对账逻辑全靠它俩字段对得上。
> 建议填写顺序:先填完开发文档第 4 章状态机,再回来填本文档(接口是从状态迁移里长出来的)。

---

## 0. 公共约定

### 0.1 Topic 命名规范
```
上行  cabin/{deviceId}/{type}        设备 → 后端
下行  cabin/{deviceId}/cmd           后端 → 设备(指令)
回执  cabin/{deviceId}/ack           设备 → 后端(指令执行结果)
```
【按你实际改。把上/下行/回执的命名规则定死,这是设备和后端的契约。】

### 0.2 消息公共字段
每条 MQTT 消息都应携带:
| 字段 | 类型 | 说明 |
|---|---|---|
| ts | long | 设备毫秒时间戳 |
| seq | long | 单调递增序号,丢包/乱序检测 |
| deviceId | string | 设备标识 |

### 0.3 HTTP 通用响应包装
```json
{ "code": 0, "msg": "ok", "data": { } }
```
【定义 code 约定:0 成功,非0 各类错误。列个错误码表。】

## 1. MQTT 上行接口(设备 → 后端)

### 1.1 摇杆姿态上报
- **Topic**: `cabin/{deviceId}/joystick`
- **方向**: ESP32 → 后端
- **频率**: ~20Hz
- **Payload**:
```json
{ "ts": 1719700000123, "seq": 10231, "x": 0.42, "y": -0.13 }
```
| 字段 | 类型 | 范围 | 说明 |
|---|---|---|---|
| x | float | [-1,1] | 已归一化、去死区的横轴量 |
| y | float | [-1,1] | 纵轴量 |

### 1.2 遥控器按键上报(W601)
- **Topic**: 【`cabin/{deviceId}/key`】
- **方向**: W601 → 后端
- **Payload**:
```json
{ "ts": ..., "seq": ..., "key": 1 }
```
【key 是原始键码还是已映射的逻辑键号?定清楚。注意:语义解释在后端,W601 只报"按了几号键"。】

### 1.3 传感器事件上报
- **Topic**: 【`cabin/{deviceId}/sensor`】
- **方向**: 传感器(ESP32) → 后端
- **Payload**:
```json
{ "ts": ..., "seq": ..., "type": "rain", "value": "..." }
```
【列出各传感器:光敏/雨滴/震动/倾斜,每种的 type 和 value 含义。设备只报客观值,不报"该不该应对"。】

### 1.4 指令执行回执 ACK ★
- **Topic**: `cabin/{deviceId}/ack`
- **方向**: 设备 → 后端
- **Payload**:
```json
{ "ts": ..., "commandId": "uuid-xxxx", "result": "done", "detail": "" }
```
| 字段 | 类型 | 说明 |
|---|---|---|
| commandId | string | 原样带回下行指令的 ID,后端据此对账 |
| result | enum | done / fail / busy ... |
【这条和下面 2.x 的下行指令是一对,字段必须能对上。】

## 2. MQTT 下行接口(后端 → 设备)★

### 2.1 设备控制指令
- **Topic**: `cabin/{deviceId}/cmd`
- **方向**: 后端 → 设备
- **Payload**:
```json
{ "commandId": "uuid-xxxx", "action": "wiper_on", "params": {} }
```
| 字段 | 类型 | 说明 |
|---|---|---|
| commandId | string | 唯一指令 ID,设备回执原样带回 |
| action | enum | 【wiper_on / door_close / alarm_on / enable_device ...枚举全】 |

### 2.2 指令清单
| action | 含义 | 目标设备 | 触发场景 |
|---|---|---|---|
| enable_device | 启用硬件 | 【】 | 【IDLE→DOOR 时】 |
| wiper_on | 开雨刮 | ESP32 | 【CRUISE 下雨应对】 |
| ... | | | |
【把所有下行 action 列全。每个标注在哪个状态/场景下会下发。】

## 3. WebSocket 下行接口(后端 → 前端)

- **连接地址**: `ws://{host}:8080/ws/attitude`(姿态)/ 【其他通道按需】
- **说明**: 前端只接收"渲染就绪"的数据,不接触原始设备量。

### 3.1 姿态帧
```json
{ "type": "attitude", "ts": ..., "pitch": 12.5, "roll": -8.2 }
```
【说明推送频率(建议后端固定频率推,与上行解耦)。】

### 3.2 事件/告警帧
```json
{ "type": "event", "event": "rain", "level": "warn", "needAck": true }
```
【前端据此展示事件提示、引导用户应对。字段按你判定逻辑定。】

### 3.3 会话状态帧
```json
{ "type": "session", "state": "CRUISE", "remainSec": 45 }
```
【告诉前端当前处于哪个阶段、剩余时间,前端据此切换 UI。这条把状态机暴露给前端。】

## 4. HTTP 接口(用户 / 会话事务)

> 这部分是你熟悉的 Controller-Service。RESTful 风格。

### 4.1 用户
| Method | Path | 说明 | 鉴权 |
|---|---|---|---|
| POST | /api/auth/register | 注册 | 否 |
| POST | /api/auth/login | 登录,返回 token | 否 |
| GET | /api/user/profile | 个人信息(里程/积分) | 是 |

### 4.2 模拟会话
| Method | Path | 说明 | 鉴权 |
|---|---|---|---|
| POST | /api/sim/start | 开始模拟,创建会话 | 是 |
| POST | /api/sim/end | 结束模拟,触发结算(**幂等**) | 是 |
| GET | /api/sim/records | 历史模拟记录列表 | 是 |
| GET | /api/sim/records/{id} | 单次记录详情(含事件明细) | 是 |
| GET | /api/sim/records/{id}/summary | AI 总结 | 是 |

【每个接口补:请求参数、响应示例、关键说明(尤其 /sim/end 的幂等如何保证)。】
